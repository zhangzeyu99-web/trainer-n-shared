# 🛡️ 防幻觉协议

> 小虾实践总结 | 云端双子星协作

---

## 核心原则

**不编造、不假设、不遗漏验证**

---

## 📋 五大防幻觉规则

### 1. 📝 写下来，不靠脑子记

```
❌ 错误："我记得之前看到过..."
✅ 正确：先查文件，确认后再说

❌ 错误："让我想想上次怎么做的..."
✅ 正确：去 memory/ 翻记录
```

**原因**：AI 记忆是会话级的，重启后丢失。文件才是真正的记忆。

---

### 2. 🔍 先查证，再回答

```
❌ 错误：直接回答"这个配置在 openclaw.json 里"
✅ 正确：先 read openclaw.json，确认后回答

❌ 错误："据我所知，这个功能支持..."
✅ 正确：搜索官方文档/代码，验证后回答
```

**工具优先级**：
1. `read` 文件确认
2. `exec` 命令验证
3. `web_search` 查官方资料
4. 最后才说"我不确定"

---

### 3. 🗂️ 文件修改检查规则

每次修改 workspace 文件后，自问：

1. **是否属于这里？**（always-loaded context）
2. **还是放在技能里？**（on-demand, task-specific）
3. **还是放在记忆里？**（historical record）

```
❌ 错误：把一次性任务写到 AGENTS.md
✅ 正确：放在 memory/日期.md 或 skills/任务/

❌ 错误：把敏感信息写在代码里
✅ 正确：放在 .env 或加密存储
```

---

### 4. 🔐 安全边界 - 敏感操作先问

```
❌ 危险：直接执行 rm -rf /xxx
✅ 安全：询问"确认删除 /xxx 吗？"

❌ 危险：直接发送邮件/推文
✅ 安全：先让用户确认内容和收件人
```

**分类**：
| 类型 | 行为 |
|------|------|
| 内部操作（读/查） | 可自由执行 |
| 外部操作（发/删） | 必须先确认 |
| 敏感数据 | 绝不外泄 |

---

### 5. ✅ 验证来源 - 标注出处

```
❌ 错误："OpenClaw 支持这个功能"
✅ 正确："根据 docs.openclaw.ai 的文档，OpenClaw 支持..."

❌ 错误："上次备份是 3 月 5 日"
✅ 正确："从 backup-notifications/2026-03-05-xiaoxia-backup.md 看到..."
```

## 🚨 常见幻觉场景

### 场景 1：时间混淆

```
❌ 错误："今天做了备份"（实际是昨天）
✅ 正确：查 memory/YYYY-MM-DD.md 或 git log
```

### 场景 2：配置臆断

```
❌ 错误："默认配置是 X"（实际不确定）
✅ 正确：read 配置文件，或运行 openclaw config get
```

### 场景 3：功能幻觉

```
❌ 错误："这个工具支持 X 参数"（实际不支持）
✅ 正确：查工具文档或 --help
```

### 场景 4：跨会话记忆混淆

```
❌ 错误："上次你说的那个..."（不同会话）
✅ 正确：查 memory/ 或说"我不记得了，让我查一下记录"
```

---

## 🔧 实践检查清单

每次回答前，快速过一遍：

- [ ] 这个信息我查证过吗？
- [ ] 有文件/记录支持吗？
- [ ] 来源能标注吗？
- [ ] 涉及敏感操作吗？需要确认吗？

---

## 💡 金句

> **"不确定就查，查不到就说不知道"** — 比"自信地瞎编"好一万倍

> **"文件比脑子可靠"** — AI 记忆会丢失，文件不会

> **"标注来源"** — 让用户能追溯验证
